**1. Write-time binding — avoid (magic number)**

```csharp
rb.AddForce(Vector2.up * 12f, ForceMode2D.Impulse);
```

`12f` is bound at the moment the code is written and never changes unless someone edits the source.
Chosen here by mistake: if jump feel needs tuning, every call site must be found and changed by hand.
The only case where this is acceptable is a genuinely universal physical constant (like `Mathf.PI`) that will never be design-adjusted.

---

**2. Compile-time binding — named constant**

```csharp
private const int MAX_JUMP_COUNT = 2;

if (jumpsRemaining > 0 && jumpsRemaining <= MAX_JUMP_COUNT)
    PerformJump();
```

`MAX_JUMP_COUNT` is bound at compile time.
Chosen because double-jump is a fixed game rule that applies to every player instance equally and is never read from a file or set in the Inspector.
A const makes the value visible to the compiler (inlined, zero overhead) and removes the magic number with a single change point.

---

**3. Load-time binding — Inspector / ScriptableObject**

```csharp
[SerializeField] private float moveSpeed = 5f;
```

`moveSpeed` is bound when Unity deserializes the scene or prefab, before the first frame runs.
Chosen because different enemy prefabs (patrol guard, boss, flying drone) each need their own speed without separate scripts.
A designer can tune values in the Inspector without touching code or rebuilding.
If the value needed to change at runtime in response to a power-up, it would move one step later: set via `moveSpeed = powerUp.speedBonus` inside a method, making binding fully runtime.
